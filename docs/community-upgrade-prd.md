# OPC圈社区粘性升级 PRD

> 版本：v1.0 | 创建：2026-05-28 | 状态：待确认
> 目标：让 opcquan 从"查询工具"变成"有回访理由的社区"

---

## 一、核心问题

用户注册后没有持续回来的理由。当前状态：
- 372 个社区信息 → 查完就走
- 广场 36 人 / 170 条帖子 → 用户之间无连接
- 有评论/点赞/通知系统但感知弱 → "有功能但用户看不到"
- 没有关注关系 → 无法形成 feed 流和社交触发

---

## 二、用户旅程（升级后）

### 旅程 A：新用户注册

```
首页/搜索 → 注册 → 填写基本信息（方向/城市/一句话介绍）
                  → 引导页："发现同城创业者"（展示 3-5 个推荐，可一键关注）
                  → 进入广场 feed → 看到内容流 → 点赞/评论
                  → 收到"xxx 回复了你"通知 → 形成首次互动闭环
```

关键时刻：
- 注册后 30 秒内必须看到"和我相关的人"（空状态 = 流失）
- 第一次互动后 24 小时内必须收到回应（通知/邮件拉回）

### 旅程 B：老用户回访

```
收到邮件/通知 → 回到网站
             → 看到"我关注的人"的新动态
             → 点赞/评论/关注新推荐的人
             → 发一条自己的进展更新
             → 被人点赞/评论 → 下次又被拉回
```

触发回访的钩子：
- 邮件："你关注的 xxx 发了新动态"
- 邮件："有人查看了你的名片"（已有）
- 站内通知角标

### 旅程 C：浏览帖子详情

```
看帖子 → 感兴趣 → 点赞/评论
      → 右侧看到"该作者的其他动态" → 点进去看更多
      → 右侧看到"同方向创业者" → 关注
      → 回到 feed 看下一篇
```

## 四、详细设计

### 1.1 关注功能

**数据模型：**

```prisma
model Follow {
  id          String   @id @default(cuid())
  followerId  String
  followingId String
  createdAt   DateTime @default(now())
  follower    User     @relation("UserFollowers", fields: [followerId], references: [id], onDelete: Cascade)
  following   User     @relation("UserFollowing", fields: [followingId], references: [id], onDelete: Cascade)

  @@unique([followerId, followingId])
  @@index([followerId])
  @@index([followingId])
}
```

User 模型新增：
```prisma
followers  Follow[] @relation("UserFollowing")
following  Follow[] @relation("UserFollowers")
```

**API：**
- `POST /api/users/[id]/follow` — 关注（幂等）
- `DELETE /api/users/[id]/follow` — 取关
- `GET /api/users/[id]/follow` — 查询是否已关注
- `GET /api/users/[id]/followers` — 粉丝列表（分页）
- `GET /api/users/[id]/following` — 关注列表（分页）

**UI：**
- 个人主页显示"关注"按钮（primary 样式，已关注变 secondary "已关注"）
- 帖子详情页作者区域显示"关注"按钮
- 右侧推荐栏的推荐卡片带"关注"按钮
- 广场名片卡不加关注按钮（空间已满，避免拥挤）
- 个人主页 tab 新增"关注 / 粉丝"计数
- "我的关注"列表页（/following）

**联动：**
- 关注时触发 Notification（type: NEW_FOLLOWER）
- 后续 feed 流按关注关系排序（Phase 2）

**不做：**
- 不做互相关注="好友"概念
- 不做关注审批
- 不做私信必须互关

### 1.2 通知系统升级

**基于现有 `notification-unread-upgrade.md` 方案落地，额外新增：**

新增通知类型：
| type | 触发场景 | 标题模板 |
|------|----------|----------|
| NEW_FOLLOWER | 被关注 | "{name} 关注了你" |
| POST_LIKED | 帖子被点赞 | "{name} 赞了你的动态" |
| POST_COMMENTED | 帖子被评论 | "{name} 评论了你的动态" |
| COMMENT_REPLIED | 评论被回复 | "{name} 回复了你的评论" |

**通知面板优化：**
- 通知项可点击，跳转到对应帖子/用户
- 未读通知有浅色背景区分
- "全部标为已读"按钮

**合并轮询（已规划）：**
- `/api/unread-summary` 统一接口
- visibilityState === 'hidden' 时暂停轮询
- 切回时立即刷新一次

**不做：**
- 不做 WebSocket 实时推送（轮询 60s 间隔够用）
- 不做推送通知（浏览器 Notification API）

### 1.3 互动可见性优化

**广场页调整：**

**首页调整：**

**帖子详情页调整：**
- 评论区折叠前默认展示 3 条（有内容感）

- 评论框提示文案："说点什么鼓励一下？"

### 2.1 帖子详情页右侧推荐栏

**布局：**
- 桌面端（≥1024px）：帖子内容左侧 2/3，右侧 1/3 推荐栏
- 移动端：推荐栏折叠到评论区下方

**推荐内容（从上到下）：**

1. **该作者的其他动态**（最近 3 条，排除当前帖）
2. **同方向创业者推荐**（同 mainTrack，排除已关注，最多 5 个）
3. **相关帖子**（同 topics 标签，最多 3 条）

**推荐卡片设计：**
- 创业者推荐卡：头像 + 名字 + 一句话 bio + "关注"按钮
- 帖子推荐卡：标题/首行内容 + 作者名 + 时间

**查询逻辑：**
```sql
-- 同方向创业者
SELECT * FROM "User"
WHERE "mainTrack" = {当前帖子作者的 mainTrack}
  AND id != {当前作者}
  AND id NOT IN (SELECT "followingId" FROM "Follow" WHERE "followerId" = {当前用户})
  AND "showInPlaza" = true
ORDER BY "lastActiveAt" DESC
LIMIT 5
```

**不做：**
- 不做算法推荐/协同过滤
- 不做"为你推荐"的个性化排序
- 规则匹配即可，后续有数据再优化

### 2.2 "创业进展"发帖类型

**数据模型：**

PostType enum 新增 `PROGRESS`

**发帖入口：**
- 广场页顶部发帖按钮旁增加引导文案："本周项目有什么进展？"
- 发帖表单中选择"类型"时，PROGRESS 类型描述："记录你的创业里程碑"

**进展帖特殊展示：**
- 卡片左侧有时间线竖条（类似 GitHub commit history）
- 可选 milestone 标签（"上线了"/"有了第一个用户"/"月收入 1k"）

**个人主页聚合：**
- 主页新增"进展"tab，按时间线展示所有 PROGRESS 类型帖子
- 像 build-in-public 记录一样呈现

**不做：**
- 不做结构化表单（周报模板/问题列表）
- 保持自由文本 + 可选标签，降低发帖门槛

### 2.3 注册后引导推荐

**触发时机：** 用户完成注册后第一次进入站点

**展示内容：**
- "发现和你方向相似的创业者"（基于 mainTrack 匹配 3-5 人）
- "你所在城市的创业者"（基于 location 匹配 3-5 人）
- 每人一个"关注"按钮

**实现方式：**
- 用户表增加 `onboardingCompleted: Boolean @default(false)` 字段
- 未完成 onboarding 时广场页顶部展示推荐条
- 关注任意 1 人后可关闭，标记 onboardingCompleted = true

**空状态兜底：**
- 如果同城/同方向无人可推荐，展示"成为第一个被推荐的人——完善你的信息"

---

### 2.4 邮件通知

**技术方案：** 复用现有 nodemailer + 腾讯企业邮箱（lib/mailer.ts 已集成，邮箱验证/密码重置已跑通）

**触发场景（按优先级）：**
| 事件 | 发送时机 | 邮件标题 |
|------|----------|----------|
| 被关注 | 即时 | "有人关注了你" |
| 帖子被评论 | 即时 | "{name} 评论了你的动态" |
| 被点赞（汇总） | 每日 | "你昨天获得了 {n} 个赞" |
| 名片被查看（汇总） | 每日 | "昨天有 {n} 人查看了你的名片" |

**退订机制：**
- User 模型增加 `emailNotifications: Boolean @default(true)`
- 每封邮件底部有退订链接

**不做：**
- 不做实时邮件轰炸（点赞汇总为每日一封）
- 不做 HTML 富邮件模板（纯文本 + 一个 CTA 按钮够了）

## 五、联动关系图

```
关注（Follow）
  ├── 触发 → 通知（NEW_FOLLOWER）
  ├── 驱动 → 右侧推荐（排除已关注）
  ├── 驱动 → 注册引导（一键关注入口）
  └── 未来 → Feed 流排序权重

通知（Notification）
  ├── 接收 ← 关注、点赞、评论
  ├── 展示 → 通知面板（铃铛）
  └── 触发 → 邮件通知（发站外）

帖子（Post）
  ├── PROGRESS 类型 → 个人主页时间线
  ├── 被点赞/评论 → 通知
  └── 被推荐 → 右侧栏

推荐栏
  ├── 数据来源 ← User.mainTrack + location
  ├── 排除 ← Follow 关系
  └── 联动 → 关注按钮
```

## 七、开发约束

### 视觉
- 所有新组件必须使用 DESIGN.md 中的语义 token（颜色、字体、间距、圆角）
- 不引入新颜色值或字体
- 交互状态完整：default / hover / active / disabled / loading
- 响应式：桌面（≥1024px）和移动端（<768px）两个断点必须覆盖
- 空状态必须有引导文案和 CTA 按钮

### 动效
- 保持与现有组件一致（transition-colors、opacity 过渡）
- 不引入新动效库
- 唯一新增：点赞按钮的心跳反馈（CSS keyframes，<5行代码）
- 骨架屏用于列表加载态

### 性能
- 列表接口必须分页（默认 20 条/页）
- 推荐栏查询加 Redis 缓存或 ISR（60s 过期）
- 通知轮询间隔 60s，后台 tab 暂停
- 图片使用 next/image 自动优化

### 代码规范
- 新 API route 统一放 `app/api/` 下
- 新组件统一放 `components/` 对应目录
- TypeScript 严格模式，不允许 any
- Prisma migration 命名：`YYYYMMDD_feature_name`

## 十、风险与兜底

| 风险 | 应对 |
|------|------|
| 用户太少，推荐栏空 | 兜底展示"热门社区"或"最近入驻" |
| 进展帖无人发 | 阿良哥自己先发 5-10 条带节奏 |
| 邮件被标垃圾 | Resend 有专用 IP，配好 SPF/DKIM |
| 通知太多烦人 | 同类型通知 24h 内合并（已有去重逻辑参考） |
| 右侧栏影响移动端 | 移动端完全隐藏或折叠到底部 |
